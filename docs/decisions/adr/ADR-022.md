# ADR-022: Package Structure and Module Boundaries

## Status
Accepted

## Context

With domain analysis, system architecture, and data modeling complete (ADR-001 through
ADR-021), docai has six well-defined pipeline components plus an LLM integration layer.
The project needs a directory layout that reflects these component boundaries, accommodates
internal complexity within components, and provides a home for shared infrastructure used
across the pipeline.

Key factors:
- Several components have significant internal complexity (Structure Extractor with three
  extraction tiers and a plugin system; Documentation Generator with two passes, context
  assembly, and hybrid follow-up; LLM connector with client abstraction, agent loop,
  tool functions, structured output, retries, and validation)
- Pydantic data models are consumed across component boundaries in a directional flow
  (Discovery → Extractor → Graph → Generator)
- Some capabilities (file I/O, hashing, caching, search/fuzzy matching) are genuinely
  cross-cutting — used by 3+ components
- The LLM agent has callable tool functions whose underlying logic overlaps with functions
  used directly by pipeline components
- Solo developer project, but internal complexity justifies structure from the start

## Options Considered

### Option A: Flat module layout
Each component is a single `.py` file under `src/docai/`. Simple to navigate, refactor
into packages later when pain is felt.

- **Pros**: Minimal structure, easy to find things, no premature organization
- **Cons**: Several components (extractor, generator, LLM) have enough internal complexity
  that single files would become unwieldy quickly. The plugin system for language configs
  needs directory structure from day one.

### Option B: Nested packages per component
Each component is a sub-package with internal modules. Shared infrastructure gets its own
package.

- **Pros**: Room for internal complexity, clear boundaries, plugin system has a natural
  home, matches the already-decided component decomposition
- **Cons**: More directories to navigate, risk of over-organization in simpler components
  (graph builder is likely just one or two modules)

### Option C: Hybrid (some flat, some nested)
Simple components stay as single modules, complex ones become packages.

- **Pros**: Right-sized structure per component
- **Cons**: Inconsistent organization, harder to predict where things live

## Decision

**Option B — nested packages per component**, with eight top-level packages:

```
src/docai/
  cli/                 # CLI entry points, orchestrator, config loading
  core/                # file I/O, hashing, cache, search/fuzzy match, path utilities
  discovery/           # file walk, detection stack, manifest + directory registry
  extractor/           # structure extraction (tiers 1-3), plugin system, language configs
  graph/               # dependency graph, bucket computation, ProcessingPlan
  generator/           # doc generation (Pass 1, Pass 2, context assembly, output)
  state/               # .docai/ management, persistence, crash recovery, locking
  llm/                 # client wrapper, agent loop, structured output, retries, tool wrappers
```

Even where a component starts small (e.g. `graph/`), the consistency of "every component
is a package" is worth more than saving one directory level. A package with two modules is
fine.

### Model placement — models live with their producer

Pydantic models live in the component that produces them:
- `ManifestEntry`, `DirectoryEntry` → `discovery/`
- `FileAnalysis`, `Entity` → `extractor/`
- `ProcessingPlan`, `WorkItem` → `graph/`
- `FileDocumentation`, entity documentation union → `generator/`
- `FileStatus` → `state/`
- `DocaiConfig` → `cli/`

Truly cross-cutting types (enums like `FileClassification`, `EntityCategory`, `FileType`,
and other shared primitives) live in `core/`.

This works because the pipeline has a clear directional flow — downstream components import
from upstream producers. Circular imports don't arise because the architecture has no
circular dependencies.

### Shared infrastructure — `core/`, not `utils/`

Cross-cutting functionality used by 3+ packages lives in `core/`. The name signals
"foundational infrastructure the rest of the project depends on" rather than the grab-bag
connotation of `utils/`. The bar for inclusion: used by multiple packages and doesn't
logically belong to any single component.

Contents:
- **File I/O**: reading files, path utilities built on the relative-path invariant
  (absolute path construction at I/O boundaries)
- **Hashing**: content hash computation (used by discovery and state)
- **Cache**: shared caching infrastructure
- **Search / fuzzy matching**: used by extractor (import resolution) and generator
  (entity reference matching) at minimum
- **Shared enums and types**: `FileClassification`, `EntityCategory`, `FileType`, and
  other cross-cutting primitives

### LLM agent tool functions — split implementation from interface

Functions the LLM agent can call (search documentation, fuzzy find, etc.) use the same
underlying logic as functions called directly by pipeline components. The split:

- **`core/`** owns the implementations — pure functions like `fuzzy_find(query, candidates,
  threshold)` that any component can import directly
- **`llm/`** owns thin tool wrappers that adapt these functions to the agent's calling
  convention (parameter schemas, structured return types, LLM-friendly error formatting)

This avoids duplicating logic while keeping the agent's tool interface co-located with the
rest of the LLM integration. Changing how the agent works means editing `llm/`, not
touching core search logic.

## Consequences

- Every component has room to grow internally without restructuring
- Model imports follow the pipeline's directional flow — easy to reason about
- `core/` has a clear admission bar (3+ consumers, no single-component home) which should
  prevent it from becoming a dumping ground
- The `llm/` package is self-contained: client abstraction, agent loop, tool wrappers, and
  internal types. Swapping providers or changing agent behavior is scoped to one package.
- Simpler components (graph, state) will have packages with only a couple of modules — this
  is acceptable overhead for consistency

- Error handling strategy across packages — not yet discussed
- Testing organization (next to code vs. separate tree) — not yet discussed
