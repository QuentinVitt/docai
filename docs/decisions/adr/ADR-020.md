# ADR-020: Documentation Output Model

## Status
Accepted

## Context
The Documentation Generator produces documentation in two passes. Pass 1 generates
purpose sentences for orientation. Pass 2 generates detailed documentation per file and
per package. We need to formalize the output data models for both passes, including the
per-entity documentation structure that the LLM produces via structured output.

This is the documentation-time counterpart to the thin extraction model (ADR-017). The
extraction model is a table of contents — five fields per entity. The documentation model
is where all richness lives: descriptions, parameters, side effects, error behavior, and
everything else the LLM judges worth documenting.

Prior decisions (ADR-004, ADR-014) established per-category required and conditional
fields. This ADR formalizes the concrete Pydantic models, revised to reflect LLM-driven
depth scaling and the simplified extraction model.

## Decisions

### Pass 1 output — Purpose sentences

```
purposes: dict[str, str]    # relative path (file or directory) -> purpose sentence
```

One combined dict for files and directories. File paths and directory paths are
distinguishable by checking against the directory registry (ADR-018). Persisted as
`.docai/purposes.json`.

### Pass 2 output — File documentation

The LLM returns structured JSON per file. The response shape depends on whether the file
has entities (source file) or not (source-like config, config file), but is represented
as a single model:

```
FileDocumentation:
  module_overview: str                                    # what the file does, always present
  entities: dict[str, EntityDocumentation] | None         # keyed by entity name, None for config files
```

For source files, `entities` contains documentation for every extracted entity (from the
FileAnalysis entity list). For source-like config and config files, `entities` is None —
the module overview is the entire documentation.

### Completeness verification

For source files, the validator checks that every entity from the FileAnalysis is
accounted for in the documentation response. An entity is accounted for if:

- It has a standalone entry in `entities`, OR
- Its name appears in a Type's `protocol_methods` or `folded_accessors` list

Missing entities trigger a validation retry with specific error feedback. Up to 3
retries before marking the file as failed.

### Complexity enum

Shared across all entity documentation types. Set by the LLM during the per-file
documentation call based on its assessment of the actual source code.

```
Complexity:
  trivial     # obvious behavior, one-liner, simple wrapper — no follow-up needed
  standard    # normal complexity, current documentation is sufficient — no follow-up needed
  complex     # non-trivial logic, subtle behavior — qualifies for per-entity follow-up call
```

Hybrid mode: after the per-file documentation call, entities marked `complex` get
individual follow-up calls with focused prompts to produce richer documentation. The
follow-up response replaces the initial documentation for that entity.

### Entity documentation — discriminated union

Per-entity documentation uses a discriminated union on `category`, matching the five
extraction categories. Each category has genuinely different documentation fields.

```python
EntityDocumentation = Annotated[
    CallableDocumentation | TypeDocumentation | MacroDocumentation |
    ValueDocumentation | ImplementationDocumentation,
    Field(discriminator="category")
]
```

The `category` value must match the entity's category from the extraction model. This
is validated during completeness checking.

### CallableDocumentation

Anything invoked to perform an action or compute a result.

```
CallableDocumentation:
  category: Literal["callable"]
  description: str                          # what it does — always present
  parameters: list[ParameterDoc] | None     # per-parameter documentation
  return_value: str | None                  # what it returns — skip for void/obvious
  side_effects: list[str] | None            # mutations, I/O, state changes
  error_behavior: str | None                # exceptions, error returns, failure modes
  example_usage: str | None                 # code example for non-obvious public APIs
  notes: list[str] | None                   # async, static, mutates self, yields,
                                            #   thread safety, language-specific concerns,
                                            #   anything important that doesn't fit above
  complexity: Complexity
```

```
ParameterDoc:
  name: str
  type: str | None                          # parameter type when available
  description: str
```

**Design rationale**: `side_effects` and `error_behavior` are separate structured fields
because they are actionable — developers scan for "does this have side effects" and "what
errors should I handle." Other behavioral properties (async, static, mutates self, yields,
thread safety) go in `notes` as a catch-all. This avoids an ever-growing list of boolean
flags and language-specific fields while keeping the most important signals structured.

### TypeDocumentation

Anything defining the shape of data or a behavior contract.

```
TypeDocumentation:
  category: Literal["type"]
  description: str                          # what this type represents — always present
  fields: list[FieldDoc] | None             # data fields, including folded accessor properties
  variants: list[VariantDoc] | None         # enum members, union variants, sealed subtypes
  relationships: list[str] | None           # extends, implements, contains — e.g. "extends BaseParser"
  protocol_methods: list[str] | None        # names of implemented dunder/protocol methods
  folded_accessors: list[str] | None        # names of accessor entities folded into fields
  example_usage: str | None
  notes: list[str] | None
  complexity: Complexity
```

```
FieldDoc:
  name: str
  type: str | None
  description: str
```

```
VariantDoc:
  name: str
  description: str
  fields: list[FieldDoc] | None             # for variants carrying data (Rust ADTs, etc.)
```

**`protocol_methods`** and **`folded_accessors`**: simple name lists, not full
documentation. They serve two purposes: (1) completeness verification — every extracted
entity is accounted for, either as standalone documentation or in these lists; (2) quick
reference — a developer sees which protocols are implemented without needing separate
entries for trivial implementations.

Protocol methods with non-obvious behavior are documented standalone as
CallableDocumentation. The LLM makes this judgment — standard `__eq__` goes in the list,
custom `__getattr__` with complex behavior gets its own entry.

**`relationships`**: structured list of inheritance/implementation/composition
relationships. Visible in signatures from extraction, but having them as a list enables
at-a-glance understanding and potential future features (inheritance diagrams).

### MacroDocumentation

Compile-time or preprocessing code generation.

```
MacroDocumentation:
  category: Literal["macro"]
  description: str                          # what this macro does — always present
  input_patterns: list[str] | None          # parameters, match arms, input patterns
  expansion_description: str | None         # what code is generated
  example_usage: str | None
  notes: list[str] | None                   # compile-time effects, feature gating,
                                            #   "what you'd write without this macro"
  complexity: Complexity
```

### ValueDocumentation

Module-level constants, variables, statics, exports.

```
ValueDocumentation:
  category: Literal["value"]
  description: str                          # what this value is and why — always present
  value: str | None                         # actual value for constants, statics, magic numbers
  notes: list[str] | None
  complexity: Complexity
```

Most values will be `complexity=trivial`.

### ImplementationDocumentation

Syntactically separate connection between a type and a behavior contract.

```
ImplementationDocumentation:
  category: Literal["implementation"]
  description: str                          # what this implementation does — always present
  connects: str                             # "Display for AST", "Iterator for TokenStream"
  notable_methods: list[str] | None         # methods that deviate from expectations
  notes: list[str] | None
  complexity: Complexity
```

### Package summary documentation

Produced for directories that qualify as packages (ADR-018).

```
PackageSummaryDocumentation:
  overview: str                             # what the directory provides, how contents relate
```

No per-entity information — package summaries are high-level overviews. The LLM receives
constituent file overviews, child package summaries, and asset listings as context
(ADR-018).

### Hybrid mode flow

1. Per-file documentation call: LLM produces `FileDocumentation` with all entities
   documented and `complexity` set per entity.
2. Collect entities with `complexity=complex` across all files in the current bucket.
3. For each complex entity: per-entity follow-up call with focused prompt, full file
   source, and dependency context. The response is a single `EntityDocumentation`
   that replaces the initial version.
4. Continue to next bucket.

### Output assembly

Final markdown output is assembled programmatically from these models:
- YAML frontmatter: from FileAnalysis + manifest data (never LLM-generated)
- Module overview: from `FileDocumentation.module_overview`
- Per-entity sections: from `EntityDocumentation` instances, formatted per category
- Markdown structure: from templates, not LLM

### Storage

- Pass 1: `.docai/purposes.json`
- Pass 2 file docs: `.docai/docs/` mirroring source tree, one file per documented file
- Pass 2 package summaries: `.docai/docs/` at the directory level
- Status tracking: `.docai/status/` mirroring source tree (per-file completion status)

All use Pydantic serialization (ADR-015).

## Consequences

- The documentation model is rich and structured while the extraction model (ADR-017)
  stays thin. Clear separation of concerns between "what exists" and "what it means."
- Per-category discriminated union gives the LLM precise JSON schemas for structured
  output. Each category gets exactly the fields relevant to its documentation needs.
- Completeness verification ensures no extracted entity is silently undocumented. Folded
  entities (protocol methods, accessors) are tracked on their parent Type.
- Complexity assessment is LLM-driven, not rule-based. The LLM sees actual source code
  and judges whether an entity needs deeper treatment.
- The `notes` catch-all field on every category prevents field proliferation while still
  capturing language-specific and behavioral concerns.
- Hybrid mode is driven by LLM complexity flags rather than entity count thresholds,
  better matching actual documentation needs.
