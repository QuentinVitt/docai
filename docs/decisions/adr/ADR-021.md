# ADR-021: Status Tracking and Configuration Model

## Status
Accepted

## Context
Two remaining data models round out the data modeling phase: generation status tracking
(how the pipeline knows what's been done and what needs work) and the configuration model
(user-facing settings in `docai.toml`). Both are operationally important but
structurally simple.

## Decisions

### Generation status tracking

Each file and package summary that enters the pipeline has a generation status. Persisted
as a single file at `.docai/status.json` — a `dict[str, FileStatus]` keyed by relative
path (files and directories share the same dict, like purposes in ADR-020).

```
FileStatus:
  status: GenerationStatus
  content_hash: str             # hash of source at time of generation
  error: str | None             # error message when failed, None otherwise

GenerationStatus:
  pending         # in the plan, not yet started
  in_progress     # LLM call dispatched, not yet complete
  complete        # documentation generated successfully
  failed          # LLM call failed after retries
```

#### Status behavior

- **`complete` + hash matches current file**: skip, documentation is up to date.
- **`complete` + hash differs**: regenerate (source changed since last documentation).
- **`in_progress`**: treated as `pending` on resume (crash recovery — the LLM call
  didn't finish, restart it).
- **`failed`**: retry on next run. The `error` field tells `docai status` why it failed
  (e.g., "context exceeded token limit", "LLM returned invalid JSON after 3 retries").
- **`pending`**: normal — work item hasn't been reached yet.

#### `content_hash` purpose

The hash on FileStatus records the file's content hash *at the time documentation was
generated*. This is compared against the current hash from the manifest (ADR-015) to
detect source changes. This is independent of the invalidation set in the ProcessingPlan
(ADR-019) — the hash check catches direct changes, the invalidation set catches
transitive dependency changes.

#### Single file storage

All statuses stored in one `.docai/status.json` file rather than per-file in a
`.docai/status/` directory. The status dict is small even for large projects (a few
hundred KB for thousands of files). Single file is simpler to load, save, and reason
about.

### Configuration model

User-facing settings in `docai.toml` (project root, committed to git). CLI flags
override config values. Config values override defaults.

```
DocaiConfig:
  project: ProjectConfig
  generation: GenerationConfig
  llm: LLMConfig
  output: OutputConfig

ProjectConfig:
  description: str | None

GenerationConfig:
  max_concurrent_calls: int              # default: 5
  context_warning_tokens: int            # default: 80000
  asset_package_threshold: int           # default: 5

LLMConfig:
  provider: str                          # default: "google-genai"
  model: str                             # default: "gemini-2.0-flash"
  api_key: str | None                    # prefer environment variable
  connection_retries: int                # default: 3
  validation_retries: int                # default: 3
  input_token_cost: float | None         # optional, for cost reporting
  output_token_cost: float | None        # optional, for cost reporting

OutputConfig:
  directory: str                         # default: ".docai/docs"
```

#### API key resolution order

1. Environment variable (primary, recommended)
2. Config file `llm.api_key` (secondary, security warning emitted)
3. Interactive prompt (fallback during `docai init`)

#### Configuration is expected to grow

This is the v1 baseline. Additional settings will be added during implementation as
needs are discovered. The Pydantic model makes adding optional fields with defaults
backward-compatible — existing `docai.toml` files continue to work when new fields
are added.

### Updated `.docai/` directory structure

With all data models finalized, the complete state directory:

```
.docai/
  version                    # state format version (plain text)
  lock                       # PID lockfile (gitignored)
  manifest.json              # dict[str, ManifestEntry] — file classifications
  directories.json           # dict[str, DirectoryEntry] — directory registry
  purposes.json              # dict[str, str] — Pass 1 purpose sentences
  graph.json                 # ProcessingPlan — ordered work item buckets
  status.json                # dict[str, FileStatus] — generation status per file
  logs/                      # detailed per-call metrics for benchmarking
  analyses/                  # FileAnalysis per processed file (mirrors source tree)
  docs/                      # generated documentation (mirrors source tree)
```

Changes from the original structure (ADR-011):
- `directories.json` added (directory registry, ADR-018)
- `status.json` replaces `status/` directory (single file instead of per-file)
- `analyses/` stores FileAnalysis models (ADR-016) per file

Gitignored: `.docai/lock`, `*.tmp` files.

## Consequences

- Status tracking is self-contained in one file. Loading project state on startup reads
  `manifest.json`, `status.json`, and `purposes.json` — three files for full pipeline
  state awareness.
- The configuration model is a Pydantic model loaded from TOML. Validation happens at
  startup — invalid config stops the pipeline before any work begins.
- The `.docai/` directory structure is fully defined. All persisted data models have
  corresponding ADRs specifying their shape and serialization.
