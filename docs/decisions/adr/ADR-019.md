# ADR-019: ProcessingPlan and Graph Builder Output

## Status
Accepted

## Context
The Graph Builder consumes dependency lists from all FileAnalysis results (ADR-016) and
the directory registry with package rules (ADR-018) to produce a ProcessingPlan — the
execution schedule for the Documentation Generator. We need to formalize the ProcessingPlan
data model and clarify what the Graph Builder persists.

## Decisions

### ProcessingPlan model

```
ProcessingPlan:
  buckets: list[list[WorkItem]]

WorkItem:
  type: WorkItemType          # file | package_summary
  path: str                   # relative file path or relative directory path

WorkItemType:
  file
  package_summary
```

The ProcessingPlan is an ordered list of buckets. Each bucket contains work items that
can be processed concurrently. Bucket ordering guarantees that by the time bucket N is
processed, all dependencies in buckets 0 through N-1 have completed documentation.

### Bucket assignment

1. Build file-level dependency graph from FileAnalysis dependency lists
2. Detect strongly connected components (cycles) via Tarjan's algorithm
3. Break cycles by placing the file with fewest cross-cycle dependencies in an earlier
   bucket
4. Compute bucket assignment via level-based topological sort:
   - Bucket 0: leaf files (no dependencies) + isolated files
   - Bucket N: files whose dependencies are all in buckets 0 through N-1
5. Schedule `package_summary` work items per ADR-018 rules. Each package summary is
   placed in the bucket after the last of its constituent files and child package
   summaries
6. Package summaries follow bottom-up directory order — child packages before parent
   packages

### WorkItem is minimal

WorkItem carries only type and path. No additional metadata. The Documentation Generator
looks up everything it needs by path — FileAnalysis for file work items, DirectoryEntry
for package summaries. The path is the universal key across all project data structures.

### No persisted dependency graph or reverse index

The Graph Builder does not persist the dependency graph or a reverse dependency index.
The forward dependency lists on FileAnalysis (ADR-016) are the source of truth for
dependency relationships. The graph is built transiently during bucket computation and
discarded.

A reverse dependency index (file → files that depend on it) was considered for
incremental invalidation but rejected. The invalidation strategy uses forward dependency
lists and bucket ordering instead (see below).

### Incremental invalidation strategy

On incremental runs, an **invalidated set** propagates through the ProcessingPlan
using bucket ordering:

1. Discovery compares stored hashes against current files. Files with changed hashes
   are added to the invalidated set.
2. Changed files get re-extracted (new FileAnalysis). The Graph Builder reruns to
   produce a new ProcessingPlan (cheap — reads dependency lists and computes buckets).
3. The Documentation Generator processes the ProcessingPlan bucket by bucket. For each
   file work item:
   - If the file is in the invalidated set → regenerate documentation, keep file in
     the invalidated set
   - If any of the file's dependencies (from FileAnalysis `dependencies`) are in the
     invalidated set → regenerate documentation, add this file to the invalidated set
   - Otherwise → use cached documentation
4. For package summary work items: if any constituent file or child package was in the
   invalidated set → regenerate the summary. Otherwise → use cached summary.

This cascades naturally through bucket order. Bucket 0 files (no dependencies) only
invalidate on hash change. Bucket 1 files check their bucket 0 dependencies against the
set. The invalidated set grows as processing proceeds, and bucket ordering guarantees
dependencies are always resolved before dependents.

No reverse index needed — forward dependency lists plus ordered processing are sufficient.

### Cycle handling

No special case for documentation context. The universal rule applies: when assembling
context for a file, use generated docs for dependencies if available, fall back to raw
source + entity directory if not. Files in a cycle that are documented first get raw
source context for their cycle partners. Files documented later get generated docs.

For bucket assignment, cycles are broken by placing the file with fewest cross-cycle
dependencies in an earlier bucket. This is a heuristic — the choice of which file
goes first affects context quality but not correctness.

### Concurrency model

One bucket at a time. All work items within a bucket are processed concurrently up to
the `max_concurrent_calls` limit (configurable in `docai.toml`, ADR-012). This is
asyncio-based — no multiprocessing needed since the bottleneck is LLM API calls, not
CPU.

### Storage

The ProcessingPlan is persisted as JSON in `.docai/graph.json`. Uses Pydantic
serialization (ADR-015). On incremental runs, the Graph Builder produces a fresh
ProcessingPlan — the stored version is overwritten, not diffed.

## Consequences

- The ProcessingPlan is a simple, flat data structure — a list of lists of work items.
  No graph representation to maintain or query.
- Incremental invalidation is handled by a runtime set that propagates through bucket
  order, not by a persisted data structure. This keeps the stored state minimal.
- The Graph Builder is a pure function: FileAnalysis dependency lists + directory
  registry + package rules → ProcessingPlan. No side state, no accumulated history.
- Cycle handling requires no special data structures — the universal context assembly
  rule and bucket ordering handle it naturally.
