# ADR-023: Error Handling Strategy

## Status
Accepted

## Context

docai is a pipeline tool where failures at different levels require different responses.
An LLM connection timeout on one file shouldn't stop the pipeline, but an authentication
failure should. Components have internal recovery mechanisms (retries, fallback strategies)
that should be exhausted before surfacing errors to the orchestrator.

The error handling strategy needs to address:
- Three distinct failure levels (component-internal, recoverable, fatal)
- A CLI tool's need for meaningful exit codes and greppable logs
- Integration with the existing status tracking model (`FileStatus` with error field)
- The reality that components use libraries (Google GenAI SDK, Pydantic, tree-sitter)
  that raise their own exceptions

Prior decisions establish that recoverable errors are logged and skipped (pipeline
continues) while pipeline errors stop execution (ADR-012). Status tracking records error
messages per file (ADR-021).

## Options Considered

### Option A: Exception hierarchy
A project-wide hierarchy with a base `DocaiError`. Components raise typed exceptions.
The orchestrator catches at the appropriate level.

- **Pros**: Natural fit for Python, aligns with how dependencies (GenAI SDK, Pydantic)
  surface errors, exception chaining (`__cause__`) preserves root cause, clean separation
  between internal and surfaced errors
- **Cons**: Exceptions as control flow can be harder to trace than explicit returns

### Option B: Result wrapper types
Components return union types (`FileAnalysis | ExtractionError`). Orchestrator
pattern-matches on results.

- **Pros**: Explicit, no hidden control flow, easier to reason about exhaustively
- **Cons**: Verbose in Python, fights against library conventions (GenAI SDK, Pydantic,
  tree-sitter all raise), requires wrapping every dependency call

## Decision

**Option A — exception hierarchy** with three error levels and per-component error types.

### Three error levels

**Level 1 — Component-internal errors.** Raised and caught within the same component.
The orchestrator never sees them. These are part of a component's internal recovery logic
and do not need to be part of the project-wide hierarchy.

Examples:
- LLM connection timeout → retry with exponential backoff (inside `llm/`)
- Tree-sitter ERROR nodes → escalate to LLM verification (inside `extractor/`)
- LLM validation failure → retry with error feedback (inside `llm/`)

Components exhaust their internal recovery before raising to the orchestrator.

**Level 2 — Recoverable errors (`ComponentError` subclasses).** A component exhausted
its internal recovery and cannot process this unit of work. The orchestrator logs the
error, marks the file as failed in status.json, and continues the pipeline.

Examples:
- All LLM retries exhausted for a file's extraction
- Validation retries exhausted — LLM keeps producing invalid output
- File cannot be read (encoding error, permissions on a single file)
- Context assembly exceeds LLM API limit

**Level 3 — Fatal errors (`PipelineError` subclasses).** The pipeline cannot continue.
The orchestrator catches these at the top level, reports to the user, and exits.

Examples:
- Invalid or unreadable configuration
- State version mismatch
- Filesystem permissions on `.docai/` directory
- Lockfile held by another process

### Authentication errors — upfront validation, not a separate error type

LLM authentication failures (invalid API key, expired credentials) are not a separate
error class. The LLM service validates credentials during construction — if the key is
invalid, the service is never created. This happens during workflow initialization,
before the pipeline starts. A failed service construction is a `ConfigError` ("API key
validation failed"), not a distinct `AuthError`.

This means the pipeline never encounters an auth error mid-run. If credentials were
somehow revoked during execution, it would surface as an LLM error, exhaust retries,
and fail files individually — but this edge case doesn't warrant a special error type.

### Exception hierarchy

```
DocaiError (base)
├── PipelineError                    # fatal, stops execution
│   ├── ConfigError                  # includes API key validation failure
│   └── StateError
└── ComponentError                   # recoverable, file-level failure
    ├── DiscoveryError
    ├── ExtractionError
    ├── GraphError
    └── GenerationError
```

`LLMError` is an internal error type within the `llm/` package — not part of the
project-wide hierarchy. When an LLM failure exhausts retries and surfaces beyond `llm/`,
the consuming component (extractor or generator) catches it and wraps it as its own
`ComponentError` subclass, preserving the `LLMError` as `__cause__`. This way the
orchestrator doesn't need to know *why* extraction failed — just that it failed. The
exception chain retains the root cause for logging and debugging.

### Error fields

All `ComponentError` subclasses carry:

| Field | Type | Purpose |
|-------|------|---------|
| `file_path` | `str` | Which unit of work failed |
| `code` | `str` | Machine-readable error code, greppable in logs |
| `message` | `str` | Human-readable description |
| `__cause__` | `Exception` | Python exception chain preserving the original error |

Error codes are uppercase strings like `EXTRACTION_PARSE_FAILED`,
`GENERATION_CONTEXT_OVERFLOW`, `LLM_RETRIES_EXHAUSTED`. These are logged alongside
the message and stored in status.json for machine-readable status tracking.

`PipelineError` subclasses carry `code` and `message` but not `file_path` (fatal
errors are not file-scoped).

### Error flow

```
Component-internal error (e.g. LLM timeout)
  → Component retries internally
  → If recovery succeeds: no error visible externally
  → If recovery exhausted: raise ComponentError subclass
      → Orchestrator catches ComponentError
      → Logs error (code + message + cause chain)
      → Records in status.json (status=failed, error=code + message)
      → Continues pipeline

Fatal error (e.g. invalid config, LLM service creation failure)
  → Raise PipelineError subclass
  → Orchestrator catches at top level
  → Reports to user
  → Exits with appropriate exit code
```

### CLI exit codes

| Code | Meaning |
|------|---------|
| **0** | Success — all files documented |
| **1** | Partial success — pipeline completed but some files failed |
| **2** | Fatal error — pipeline could not complete |

Partial success (exit code 1) is distinct from fatal failure (exit code 2) so that
scripts and CI can differentiate between "3 out of 200 files had issues" and "couldn't
run at all."

## Consequences

- Components own their internal recovery logic — the orchestrator doesn't need to
  understand retry semantics or fallback strategies
- Recovery policy is centralized in the orchestrator. Adding a `--strict` mode (any
  failure aborts) requires changing only the orchestrator's catch behavior.
- Per-component error types allow component-specific context without the orchestrator
  needing to special-case different failure modes
- Error codes provide greppable logs and machine-readable status without parsing messages
- The wrapping pattern (`LLMError` → `ExtractionError` with `__cause__`) keeps component
  boundaries clean while preserving diagnostic detail
- Auth errors are caught before the pipeline starts via LLM service construction,
  eliminating the need for mid-pipeline auth error handling

## Open Questions

- Specific error code catalog per component — define during implementation as failure
  modes are discovered
