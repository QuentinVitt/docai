# ADR-025: Testing Strategy

## Status
Accepted

## Context

docai has a natural split between deterministic logic (detection stack, tree-sitter
extraction, graph computation, context assembly, config merging, state management) and
non-deterministic LLM-dependent code (Tier 3 extraction, Pass 1/2 documentation
generation). The testing strategy needs to cover the deterministic core thoroughly while
handling LLM integration pragmatically.

The project processes source code across multiple languages, so representative test inputs
are essential. The LLM client uses a thin wrapper interface (`send`, `send_structured`)
that provides a natural mocking seam.

## Decision

### Test organization — separate tree mirroring package structure

```
tests/
  test_core/
  test_discovery/
  test_extractor/
  test_graph/
  test_generator/
  test_state/
  test_workflow/
  fixtures/
```

Tests live in a separate `tests/` directory rather than co-located with source code. This
keeps test code out of the installable package and provides a natural top-level home for
`fixtures/`.

### What to test

**High-value, test thoroughly:**
- Discovery detection stack (magic bytes, extension map, shebang, filename map, docaiignore
  pattern matching, classification precedence)
- Tree-sitter entity extraction and import resolution (Tier 1) — per-language correctness
  of entity mapping tables, import pattern tables, resolution heuristics
- Graph builder bucket computation — dependency ordering, cycle detection and breaking,
  package summary scheduling, bottom-up evaluation
- Context assembly in the generator — dependency doc lookup, fallback chain, entity
  reference matching
- Config merging — field-level override behavior across all four precedence layers
- State management — atomic writes, crash recovery (resuming from `in_progress`),
  incremental invalidation set propagation
- Validation logic — LLM response validation, completeness verification (all entities
  accounted for), retry-with-feedback behavior

**Test the structure, not the output:**
- LLM-dependent code paths: test that correct context is assembled and passed to the LLM
  client, that valid responses are processed correctly, that invalid responses trigger
  retries, that retry exhaustion raises the right `ComponentError`. Do not test LLM output
  content — that's a benchmarking concern.

**Not worth testing:**
- CLI argument parsing (argparse handles this)
- Output markdown formatting (visible immediately when running the tool; snapshot tests
  at most)
- Rich progress display

### Fixture strategy — scenario-based mini projects

Test fixtures are organized by scenario, not by language. Each fixture is a self-contained
mini project directory that exercises specific behavior:

```
fixtures/
  small_python_project/      # 5-6 files, full pipeline end-to-end
  circular_deps/             # import cycles, tests graph builder cycle handling
  tier2_imports/             # SCSS/HTML/Makefile with imports, tests heuristic tier
  mixed_languages/           # Python + JS + Rust, tests multi-language handling
  error_cases/               # malformed files, unreadable files, binary as source
  tree_sitter_errors/        # valid code with parse edge cases
```

Tests point at fixtures and assert on outputs. This is more maintainable than inline
source code strings and allows fixture reuse across test levels.

### LLM calls in tests — mock the wrapper

Unit and integration tests mock the LLM client at the wrapper interface (`send`,
`send_structured`). Mocks return canned responses that test the code's handling of LLM
output without calling the API. This covers: valid response processing, invalid response
handling, retry behavior, error escalation.

End-to-end output quality validation is a separate benchmarking activity. Run against
real fixtures with real LLM calls, inspect output manually or via snapshot comparison.
This does not run in CI — it runs when prompts or context assembly logic change and the
developer wants to assess impact. The existing `.docai/logs/` infrastructure supports
this.

## Consequences

- Deterministic pipeline logic gets thorough coverage — these are the highest-risk areas
  for subtle bugs (wrong bucket order, missed invalidation, broken import resolution)
- LLM-dependent code is tested for structural correctness without API coupling
- Scenario-based fixtures provide realistic inputs without maintaining per-language test
  suites
- Separate test tree keeps the installable package clean
- Output quality is a benchmarking concern, not a CI concern — avoids flaky tests from
  non-deterministic LLM responses
