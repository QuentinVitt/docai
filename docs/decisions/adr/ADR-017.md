# ADR-017: Entity Data Model

## Status
Accepted

## Context
The Structure Extractor produces an entity directory per source file as part of the
FileAnalysis (ADR-016). Entities are the core unit of documentation — every documented
function, class, constant, etc. is an entity. We need to formalize the extraction-time
entity data model: what the extractor captures from source code for downstream consumption
by the Graph Builder and Documentation Generator.

Prior decisions (ADR-004, ADR-014) established six entity categories with per-category
kinds and extensive per-category field lists. This ADR revisits those decisions in light
of the extraction vs. documentation model separation: the extraction model should be as
thin as possible, capturing only what downstream components need programmatically. All
richness — descriptions, side effects, error behavior, complexity judgments — belongs in
the documentation model, produced by the LLM during documentation generation when it has
the full source code.

## Decisions

### Single flat Entity model

No discriminated union, no inheritance hierarchy. All entities share the same five fields
regardless of category. The extraction model serves as a **table of contents** for the
file — here's what exists, what it's called, and roughly what it is. The Documentation
Generator receives this table of contents plus the full source code and produces rich
documentation.

```
Entity:
  category: EntityCategory    # callable | macro | type | value | implementation
  name: str                   # entity name as it appears in code
  kind: str                   # specific variant — freeform string
  parent: str | None          # name of enclosing type for depth-1 entities, None for top-level
  signature: str | None       # full signature/declaration as it appears in source
```

A discriminated union with per-category subclasses was considered but rejected. With no
per-category fields on the extraction model, every entity has the same shape. The union
would add modeling complexity (base class, five subclasses, discriminator setup) with no
structural benefit.

### EntityCategory enum

```
EntityCategory:
  callable
  macro
  type
  value
  implementation
```

Five values. This is a closed set — the categories map to documentation templates in the
Documentation Generator. Module is excluded (implicit, created during documentation
generation per ADR-014).

### Kind as freeform string

`kind: str` — not a per-category enum. Rationale:

- The tree-sitter extraction path produces well-defined kinds from the AST node mapping
  table. These are correct by construction and don't need enum validation.
- The LLM extraction path (Tier 3 fallback) produces whatever kind label best describes
  the entity for that language. Constraining to a fixed enum with `other` loses
  information (e.g., `companion_object` flattened to `other`).
- No downstream component branches programmatically on specific kind values. The kind is
  a label passed through to the Documentation Generator as a hint.

Convention: lowercase, underscore-separated (e.g., `function`, `method`, `constructor`,
`class`, `struct`, `enum`, `constant`, `trait_impl`). Language configs and LLM prompts
follow this convention for consistency.

### Signature field

`signature: str | None` — the full function signature, type declaration header, or
equivalent as it appears in source code. Examples:

- Callable: `def parse_file(path: str, strict: bool = False) -> AST:`
- Type: `class Parser(BaseVisitor):`
- Value: `MAX_RETRIES: int = 3`
- Macro: `#define MAX(a, b)`
- Implementation: `impl Display for AST`

Optional because some entity types in some languages lack a meaningful one-line signature.
Serves two downstream purposes:

1. **Context assembly**: when the Documentation Generator assembles context for a file's
   dependencies, it includes entity signatures from dependency files. This gives the LLM
   the interface at a glance without needing full source of every dependency.
2. **Documentation prompting**: the LLM sees the signature alongside the entity name and
   category as part of the entity directory.

### Extraction depth rule

Entities are extracted at two levels of nesting:

- **Depth 0** — all top-level entities in a file: functions, classes, constants, type
  aliases, macros, etc.
- **Depth 1** — entities that define a type's interface: methods, constructors, properties,
  enum variants, static members, associated types/constants. These carry a `parent` field
  referencing the enclosing type.

Depth 1 extraction is scoped to **type members**, not arbitrary nesting. A method inside
a class is extracted because it's part of the class's API. A function inside another
function is not extracted because it's an implementation detail of the enclosing function.

Everything else is not extracted as a separate entity:
- Variables inside functions or methods
- Functions defined inside other functions
- Classes defined inside functions
- Any nesting beyond depth 1 (nested classes inside classes, closures inside methods)

These are documented as part of their parent's description when the LLM sees the full
source code during documentation generation.

### Accessor handling

Accessors (getters, setters, properties) are extracted as regular entities with
`category=callable` and an appropriate kind (e.g., `getter`, `setter`, `property`).
They are not folded into the parent Type at extraction time. The Documentation Generator
decides presentation — folding trivial accessors into the parent Type's field
documentation, or documenting complex ones standalone — based on the actual source code.

This moves the complexity/triviality judgment from extraction (where it's hard and
error-prone across languages) to documentation generation (where the LLM sees the code
and can make nuanced decisions).

### Depth scaling — LLM-driven

The extraction model carries no complexity signals (no line counts, no role, no
decorators, no generic type parameter flags). Depth scaling is fully delegated to the
LLM during documentation generation:

- The per-file documentation call produces documentation for all entities. The LLM
  determines depth based on the actual source code — it can see visibility, decorators,
  generics, code complexity, and everything else directly.
- The LLM flags entities that need deeper treatment (see hybrid mode below).
- Kind and category provide documentation template guidance, not depth rules.

### Hybrid mode trigger

The per-file documentation call always happens first. To determine whether per-entity
follow-up calls are needed, the LLM includes a `needs_deep_docs` flag per entity in its
response. Entities flagged by the LLM as needing deeper treatment get individual
follow-up calls with focused prompts. This replaces the previous threshold-based approach
(entity count > 15) with an LLM-driven decision that accounts for actual complexity
rather than entity count.

### Dropped fields (relative to ADR-014)

- **`role`**: visibility and semantic intent (public, private, internal, protocol,
  accessor) are visible in the source code. The LLM determines documentation depth and
  accessor folding directly from the code during documentation generation. No downstream
  component branches programmatically on role.
- **`decorators`**: visible in the source code. Were kept as a depth scaling signal, but
  depth scaling is now fully LLM-driven. Adding decorators to extraction increases
  failure surface (LLM must list them correctly for every entity) without downstream
  payoff.
- **`line_start` / `line_end`**: noisy complexity proxy, extraction failure risk (LLMs
  count lines poorly), no context-slicing use since full file source is always provided.
- **Per-category fields**: the extraction model is a table of contents, not a
  documentation structure. All per-category richness (parameters, return values, side
  effects, error behavior, fields, variants, etc.) is produced by the Documentation
  Generator, not the extractor.
- **Per-category kind enums**: kind is a freeform label. No downstream component
  validates or branches on specific kind values.

## Consequences

- The entity list in FileAnalysis is `list[Entity]` — one flat Pydantic model, simple to
  serialize, iterate, and reason about.
- The extraction model is minimal (five fields). This reduces LLM extraction failure
  surface — fewer fields to get wrong per entity, especially important for files with
  many entities.
- All documentation richness lives in the documentation output model (separate from this
  ADR). The Documentation Generator owns complexity judgments, depth decisions, and
  presentation choices.
- Hybrid mode (per-entity follow-up calls) is triggered by LLM judgment rather than
  entity count thresholds, better accounting for actual complexity.
- The documentation output model (what the LLM produces) will be formalized in a
  subsequent ADR. That model carries the per-category fields from ADR-014 (descriptions,
  side effects, error behavior, parameters, fields, variants, etc.) as its domain.
