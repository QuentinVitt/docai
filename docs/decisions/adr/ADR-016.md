# ADR-016: FileAnalysis Data Model

## Status
Accepted

## Context
The Structure Extractor produces a FileAnalysis for every file that enters the extraction
pipeline. This is the primary data structure consumed by the Graph Builder (for dependencies)
and the Documentation Generator (for entities and file type). We need to formalize its shape,
determine which fields earn their place on the persisted model, and clarify ownership
boundaries with the manifest (ADR-015).

## Decisions

### FileAnalysis fields

```
FileAnalysis:
  file_path: str                # relative path, matches manifest key
  file_type: FileType           # what the extractor determined this file to be
  entities: list[Entity]        # empty for source-like config and config files
  dependencies: list[str]       # resolved relative paths to project files
```

### FileType enum

Replaces the `subcategory` concept from ADR-014. This is the extractor's own determination
of what the file is, based on what it found — not a refinement of Discovery's classification.

```
FileType:
  source_file         # programming language, full entity extraction performed
  source_like_config  # not a programming language, but imports to project files found
  config_file         # not a programming language, no imports found
```

`FileType` lives exclusively on FileAnalysis. The manifest (ADR-015) does not carry it.
Discovery and the extractor each own their respective data — no write-back coupling.

### Dependencies

`list[str]` — resolved relative paths to other project files. No metadata about which
entities are referenced; that matching happens downstream in the Documentation Generator
during context assembly (lowercase entity name search in file source).

### Entity model

Defined separately in ADR-017. The `entities` list contains the full entity directory
for source files and is empty for source-like config and config files.

### Dropped fields

- **`raw_imports`**: purely diagnostic. If import resolution bugs need debugging, add
  logging in the extraction path rather than persisting per file.
- **`parse_errors`**: diagnostic. Tree-sitter ERROR node handling (LLM verification) is
  resolved within the extractor before producing the FileAnalysis. No downstream component
  branches on this field. Log during extraction instead.
- **`extraction_method`**: diagnostic. No downstream component checks whether extraction
  used tree-sitter, LLM fallback, or heuristics — they all consume the same FileAnalysis
  contract. Log during extraction instead.

### Self-containedness

`file_path` is stored on the model even though it's also derivable from the storage path
(`.docai/analyses/` mirrors the source tree). A loaded FileAnalysis should be
self-describing without relying on external context.

### Force-included unknown files that fail

If a force-included unknown file cannot be read (permissions, binary content, encoding
error) or LLM language identification fails, no FileAnalysis is produced. The pipeline
logs a warning and moves on. The file remains in the manifest as
`classification=unknown, override=include` with no corresponding FileAnalysis.

### Storage

Persisted as JSON in `.docai/analyses/`, mirroring the source tree structure.
One file per FileAnalysis. Uses Pydantic serialization (ADR-015).

## Consequences

- FileAnalysis is a minimal four-field model. All diagnostic information lives in
  extraction-time logs, not in the persisted data structure.
- The Graph Builder reads `dependencies` from all FileAnalysis results to build the
  project-wide dependency graph.
- The Documentation Generator reads `file_type` to determine documentation depth
  (entity-level vs. module-level only) and `entities` for the entity directory.
- No coupling between the manifest and FileAnalysis — each component owns its own output.

## Open Questions
