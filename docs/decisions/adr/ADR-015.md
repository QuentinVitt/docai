# ADR-015: File Manifest Data Model

## Status
Accepted

## Context
File Discovery produces a manifest consumed by all downstream components. We need to
formalize the data representation: what each entry carries, how classification and
subcategorization interact, how user overrides are represented, and which modeling library
underpins all project data structures.

Prior decisions (ADR-005, ADR-007, ADR-014) established five file classifications, three
processed subcategories, and the `.docaiignore` override mechanism. This ADR pins down the
concrete data model.

## Decisions

### Modeling library: Pydantic

All project data structures use **Pydantic v2** models. Rationale:

- **LLM structured output**: Pydantic's `.model_json_schema()` generates the JSON schemas
  sent to the GenAI API for structured responses. Internal models *are* the LLM response
  schemas — no parallel schema layer needed.
- **Disk persistence**: the entire `.docai/` state directory is JSON. Pydantic's
  `.model_dump_json()` / `.model_validate_json()` handles serialization cleanly.
- **Validation**: runtime validation of LLM responses and loaded state files with clear
  error messages.

### Manifest entry fields

Each entry in the manifest represents one file. The manifest is a flat
`dict[str, ManifestEntry]` keyed by **relative path** (string, relative to project root).
The path is the dict key, not stored on the entry.

```
ManifestEntry:
  classification: FileClassification    # what Discovery's detection stack determined
  language: str | None                  # None for assets, docs, ignored, unknown
  content_hash: str | None              # SHA-256, only for files entering the extractor
  override: FileOverride | None         # user override from .docaiignore, None if no override
```

### Enums

```
FileClassification:
  processed       # enters Structure Extractor
  documentation   # existing project docs (README, etc.)
  asset           # binary/resource files
  ignored         # invisible, not mentioned in output
  unknown         # unrecognized, warned, skipped

FileOverride:
  include    # force-included via ! pattern in .docaiignore
  exclude    # force-excluded via .docaiignore pattern
```

### Classification vs. override semantics

`classification` records what Discovery's detection stack determined — it is a permanent
fact about the file's identity. `override` records user intent from `.docaiignore` — it
changes what the pipeline *does* with the file, not what the file *is*.

- A force-included unknown file keeps `classification=unknown`. The override triggers
  LLM identification. If successful, `language` is populated and the file enters the
  extractor. Classification does not change.
- A force-excluded `.py` file keeps `classification=processed`. The override prevents
  it from entering the extractor. It receives no `content_hash`.

### Which files get hashed

Files that actually enter the Structure Extractor receive a `content_hash`. This is:
- Files with `classification=processed` and `override != exclude`
- Files with `override=include` that are successfully identified by the LLM

All other files have `content_hash=None`.

### Language field

`str | None`. Known languages are stringly-typed (e.g., `"python"`, `"rust"`,
`"javascript"`). The extraction method map matches on these strings to find the right
extraction strategy. LLM-identified languages for force-included unknowns follow the same
convention. No language enum — the LLM fallback path can produce any language name, and
maintaining a comprehensive enum adds overhead without safety benefit.

### Manifest container

The manifest is `dict[str, ManifestEntry]` with no wrapper model. Aggregate counts
(files per classification) are computed on the fly when needed — they are cheap to derive
and storing them would create staleness risk when entries are updated.

### Dropped fields

- **`detection_method`**: purely diagnostic, not consumed by any pipeline stage. If needed
  for debugging, add logging in the detection stack rather than persisting per entry.
- **`file_size`**: no downstream consumer identified. Trivial to add later if needed.

## Consequences

- All downstream ADRs for data structures use Pydantic v2 models.
- The manifest is the single source of truth for "what files exist and what are they."
  No component re-walks the filesystem — they consult the manifest.
- The manifest owns only what Discovery knows. File type as determined by extraction
  (source file, source-like config, config file) lives on the FileAnalysis model,
  not the manifest. No write-back coupling between extractor and manifest.
- Override semantics are clean: classification is identity, override is user intent.
  Pipeline logic checks both fields to determine behavior.

## Open Questions
None.
