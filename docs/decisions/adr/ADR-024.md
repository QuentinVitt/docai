# ADR-024: Configuration Loading, Logging, and Display Strategy

## Status
Accepted

## Context

docai needs a configuration loading strategy that merges four sources (defaults, config
file, environment variables, CLI arguments), a logging approach used consistently across
all components, and a display strategy for progress bars that doesn't conflict with log
output.

Prior decisions establish: `docai.toml` as the config file format (ADR-012), `DocaiConfig`
as the Pydantic model (ADR-021), precedence order defaults < config < env < CLI (ADR-012),
`rich` for progress reporting (ADR-012), and verbosity flags `-v` / `-q` (ADR-012).

Additionally, the introduction of a `workflow/` package (ADR-022) separates pipeline
orchestration from CLI concerns, raising the question of how workflow progress is
communicated to the user.

## Decision

### Configuration loading

**Merge strategy: field-level, not section-level.** Each layer in the precedence chain
overrides individual fields, not entire sections. `--model gemini-2.0-pro` on the command
line overrides the model but preserves all other `[llm]` settings from the config file.

**Loading sequence:**
1. Construct defaults (hardcoded in the `DocaiConfig` model via Pydantic `Field(default=...)`)
2. Read and parse `docai.toml` if it exists → deep merge over defaults
3. Read recognized environment variables (`DOCAI_API_KEY`, etc.) → deep merge over result
4. Apply CLI argument overrides → deep merge over result
5. Construct `DocaiConfig` from the merged dict → Pydantic validates types and constraints
6. Semantic validation (e.g. `max_concurrent_calls > 0`, output directory not inside
   `.docai/`) — fail fast with `ConfigError`

Validation happens at load time. If configuration is invalid, the pipeline never starts.
The only validation that happens later is LLM service construction (API key check), which
occurs during workflow initialization and surfaces as `ConfigError` if it fails (ADR-023).

**CLI flags are selective.** Only frequently changed settings are exposed as CLI flags:
model, output directory, `--force`, `--yes`, `-v`, `-q`, file/directory path argument.
Rarely changed settings (retry counts, token costs, asset package threshold) are
config-file only. This keeps `--help` clean.

**`DocaiConfig` lives in `cli/`.** The model definition and the loading/merging logic are
both CLI concerns — the config file format, env var names, and CLI flag mapping are all
presentation-layer decisions. Downstream components receive the constructed config (or
relevant subsections) as a dependency, not the raw TOML.

### Workflow package

Pipeline orchestration logic lives in `workflow/`, separate from `cli/`. This separates
two concerns:

- **`cli/`**: argument parsing, user interaction (prompts, confirmation), config loading,
  log/display setup, translating commands into workflow invocations
- **`workflow/`**: pipeline sequencing, inter-step validation, component coordination

Each CLI command maps to a workflow. `docai generate` invokes the generate workflow.
`docai status` invokes a read-only status workflow. The workflow contains the logic for
"create LLM service → run discovery → run extraction → build graph → generate docs"
including the inter-step validation (e.g. verifying the LLM service was created
successfully before entering the pipeline).

Updated package structure (amends ADR-022):

```
src/docai/
  cli/                 # argparse, user interaction, config loading, log/display setup
  core/                # file I/O, hashing, cache, search/fuzzy match, shared types
  discovery/
  extractor/
  graph/
  generator/
  state/
  llm/
  workflow/            # pipeline orchestration, inter-step validation, workflow definitions
```

### Logging strategy

**Standard library `logging` is the primary output mechanism.** Every component logs at
the appropriate level using module-level loggers (`logger = logging.getLogger(__name__)`).
Components never import `rich`, never know about verbosity flags, never manage their own
output. They just log.

**Verbosity mapping:**

| Flag | Log level | What the user sees |
|------|-----------|--------------------|
| `-v` | DEBUG | Per-file detail, LLM call metadata, extraction tier decisions |
| *(default)* | INFO | Stage progress, file type counts, warnings |
| `-q` | WARNING | Warnings and errors only |

**File logging:** A `logging.FileHandler` writes to `.docai/logs/` for persistent run
records. This captures all log levels regardless of the console verbosity setting,
providing full diagnostic detail for debugging without cluttering terminal output.

### Display strategy — shared Rich console

**One `rich.console.Console` instance, created once in `cli/` during initialization.**
Both the logger and progress bars use this same console to prevent output conflicts
(garbled progress bars, overwritten log lines, cursor position issues).

- The root logger gets a `rich.logging.RichHandler` backed by the shared console
- Progress bars (via `rich.progress`) use the same shared console
- Rich coordinates output — log lines render above the progress bar, the bar redraws
  cleanly

**All output goes to stderr.** Logging and progress display write to stderr. Since docai
writes documentation to files (not stdout), this keeps stdout clean for potential future
piping or structured output.

**Progress bars are a thin, narrowly scoped UI layer.** They exist only for the two or
three hot loops with concurrent work (extraction progress, generation progress, possibly
Pass 1). The workflow drives progress updates through a lightweight callback or progress
object — this is the only bridge between workflow and display. Everything else flows
through standard logging.

## Consequences

- Field-level merging means CLI flags are surgical overrides, not section replacements
- Fail-fast validation catches config errors before any pipeline work begins
- Components are logging-only — no display coupling, no verbosity awareness
- The shared console eliminates Rich/logging output conflicts
- File logging provides full diagnostic history regardless of terminal verbosity
- The `workflow/` package cleanly separates "what the pipeline does" from "how the user
  interacts with it"
- Selective CLI flag exposure keeps the interface clean while TOML covers all settings

## Open Questions

- Exact list of recognized environment variables beyond `DOCAI_API_KEY` — define during
  implementation
- Progress callback interface between workflow and CLI — exact shape to be determined,
  but intentionally minimal (stage started, item completed, stage completed)
