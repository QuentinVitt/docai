# ADR-018: Directory Registry and Package Rules

## Status
Accepted

## Context
Package summaries provide entry-point overviews for directory-level documentation. We need
to formalize how directories are represented, which directories qualify as packages, and
how the package decision propagates through the directory tree. The Graph Builder uses these
rules to schedule `package_summary` work items in the ProcessingPlan.

Prior decisions established that package summaries are generated for directories with
sufficient documented content (ADR-009, ADR-010, ADR-014). This ADR pins down the concrete
rules and data model.

## Decisions

### Directory registry

Discovery produces a directory registry alongside the file manifest (ADR-015). The
directory registry is a separate structure — `dict[str, DirectoryEntry]` keyed by relative
directory path. The file manifest stays file-only.

```
DirectoryEntry:
  files: list[str]              # relative paths of direct child files
  subdirectories: list[str]     # relative paths of direct child directories
  asset_summary: AssetSummary | None  # aggregated asset info, None if no assets
```

```
AssetSummary:
  count: int                    # total number of asset files in this directory
  types: dict[str, int]         # extension -> count, e.g. {"png": 12, "svg": 8}
```

The directory registry captures only direct children — no recursive subtree information.
The directory tree structure is implicit in the nesting of paths.

Only non-ignored directories are included. Pruned directories (`.git/`, `node_modules/`,
etc.) never enter the registry.

### Two package types

#### Normal package

A directory qualifies as a normal package if it contains **2 or more documentable items**
as direct children. Documentable items are:

- Processed files (direct children with `classification=processed` in the manifest)
- Child normal packages
- Child asset packages

These count uniformly — 1 processed file + 1 child package = 2 documentable items =
normal package. This keeps the rule simple and handles mixed directories naturally.

Examples:
- Directory with 2+ processed files (any combination of source, source-like config,
  config) → normal package
- Directory with 2+ child packages → normal package
- Directory with 1 child package + 1 processed file → normal package
- Directory with 1 processed file + 0 child packages → not a package

#### Asset package

A directory qualifies as an asset package if:

- It contains **only** asset files (plus ignored, documentation, or unknown files — no
  processed files)
- It has **no child packages**
- The number of asset files meets a configurable threshold (default: 5)

Asset packages get a summary describing the asset collection (e.g., "50 SVG icons covering
navigation, status indicators, and social media logos").

### Bottom-up evaluation

Package qualification is evaluated **bottom-up** — leaf directories first, then their
parents. This is necessary because a directory's package status depends on whether its
children are packages.

Evaluation order:
1. Evaluate all leaf directories (no subdirectories) for normal package and asset package
   rules
2. Move up one level. Each directory now knows which of its children are packages. Evaluate
   using the same rules — child packages count as documentable items.
3. Repeat until the project root is reached.

This means package status propagates upward naturally. When two sibling directories each
become packages, their parent becomes a package too (2 documentable child items). The
documentation hierarchy grows organically with the project structure.

### Edge cases

**Directory with 2 child directories, each containing 1 processed file:**
Neither child qualifies as a package (1 processed file each). The parent has 0 direct
processed files and 0 child packages → not a package. This is correct — the files are
documented individually, and the directory structure reflects intentional separation into
distinct concerns.

**Directory with only documentation files (READMEs, etc.):**
Documentation files are not processed files and not assets → the directory is neither a
normal package nor an asset package. The documentation files get Pass 1 purpose sentences
and appear in any parent package summary, but the directory itself doesn't get its own
package summary. This avoids circular reasoning — the directory's summary would be based
on documentation that already exists.

**Deeply nested asset directory:**
`src/components/icons/` with 50 SVGs qualifies as an asset package. Its parent
`src/components/` counts it as a documentable child item. If the parent has other
documentable items (processed files or other child packages), it becomes a normal package
whose summary mentions the icons subdirectory.

**Single processed file + many assets:**
A directory with 1 processed file and 20 asset files has only 1 documentable item (the
processed file — assets don't count as documentable items for normal package threshold).
Not a normal package. Not an asset package either (has a processed file). The file's own
module documentation covers the relationship with the assets. This matches the intuition
that the assets are likely imported by or related to the single file.

**Project root:**
The project root is evaluated like any other directory. If it qualifies as a package, it
gets a package summary. For most non-trivial projects it will, since it typically contains
multiple processed files or child packages.

### Package summary context

When the Documentation Generator produces a package summary, it receives:

- Pass 1 purpose sentences for all direct child files and directories
- Module overviews for all documented child files (processed files with completed docs)
- Child package summaries (for child packages that have been documented — guaranteed
  available due to bottom-up scheduling)
- Asset summary from the DirectoryEntry (filenames and type counts for asset files)

### Configuration

The asset package threshold is configurable in `docai.toml`:

```toml
[generation]
asset_package_threshold = 5    # minimum asset files for asset package qualification
```

## Consequences

- Discovery produces both a file manifest and a directory registry. Two separate
  structures, each with a clean purpose.
- Package qualification is deterministic and rule-based — no LLM calls needed to decide
  whether a directory gets a summary.
- The bottom-up evaluation ensures package status propagates naturally through the
  directory tree. Adding files to a project can cause new packages to emerge at multiple
  levels.
- The Graph Builder uses these rules to schedule `package_summary` work items in the
  ProcessingPlan, ordered bottom-up (child packages before parent packages).
- Normal packages and asset packages get different documentation treatment — normal
  packages include entity overviews and dependency context, asset packages describe the
  collection.
